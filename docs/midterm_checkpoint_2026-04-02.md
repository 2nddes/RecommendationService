# RecommendationService 中期检查文档

- 检查日期: 2026-04-02
- 项目阶段: 中期（核心链路已打通，局部能力待增强）
- 检查范围: 服务架构、推荐链路、训练运维、容错与可观测性、待办缺口

## 1. 当前已完成内容（做了哪些）

### 1.1 服务框架与基础能力

1. 已完成 Flask 微服务主干，提供统一 API 前缀与健康检查。
2. 已实现统一错误处理与统一响应结构（code/message/data）。
3. 已接入日志系统（RotatingFileHandler，文件轮转），支持通过配置调整日志级别与路径。
4. 已支持内部鉴权开关（INTERNAL_SECRET），可在内网模式与受控模式之间切换。

### 1.2 推荐核心链路（Recall -> Rank -> Rerank）

1. 已完成推荐流水线编排 RecommendationPipeline，形成固定三阶段执行链路。
2. 召回阶段已落地 Two-Tower 召回（在线构建用户向量 + 离线物品向量索引检索）。
3. 排序阶段已落地 MMoE 精排（多任务联合打分，click/collect/comment/rating）。
4. 重排阶段已接入 RandomShuffle 作为当前策略（可控随机扰动，增强多样性）。
5. 已对外提供核心接口:
   - GET /api/v1/recommend/user
   - GET /api/v1/recommend/item
   - GET /api/v1/recommend/trending

### 1.3 训练与模型运维

1. 已完成训练触发与任务管理接口:
   - POST /api/v1/admin/train
   - GET /api/v1/admin/tasks
   - GET /api/v1/admin/tasks/<task_id>
   - GET /api/v1/admin/status
   - POST /api/v1/admin/refresh
2. 已实现训练任务异步执行（线程任务管理）与数据库作业状态同步（pending/processing/completed/failed）。
3. 已完成模型训练分发逻辑，可按 component/model 定向触发:
   - ranking/mmoe
   - ranking/xgb
   - recall/two_tower
4. 已完成 Artifact 持久化能力，支持记录与读取最新训练产物路径。

### 1.4 Two-Tower 召回与索引能力

1. 已完成 Two-Tower 训练逻辑（正负样本、BPR 优化、批训练、离线向量输出）。
2. 已支持物品向量落盘与 HNSW 索引构建。
3. 已支持索引缓存、文件变更自动刷新、回退到 numpy 检索（无 hnswlib 时）。
4. 已实现启动阶段自动全量构建与周期性刷新任务（可配置开关与间隔）。

### 1.5 MMoE 精排能力

1. 已完成 MMoE 模型结构与离线训练主流程。
2. 已完成多任务标签构造与训练可用性校验。
3. 已在推理阶段接入多源特征（用户画像、短期行为、长期兴趣、影片统计、标签特征等）。
4. 已支持模型 bundle 化保存（参数 + 索引映射 + 归一化统计），保证训推一致性。

### 1.6 RAG 推荐能力

1. 已上线流式 RAG 接口 POST /api/v1/recommend/rag/stream（SSE: start/movie/done）。
2. 已支持基于 MySQL 电影数据构建 LangChain + FAISS 向量索引。
3. 已支持强制重建索引（rebuild_index）与本地索引加载/回建。

## 2. 已完成内容的实现方法（怎么做的）

### 2.1 架构实现方式

1. 采用模块化分层:
   - api: 对外协议层
   - reco: 召回/排序/重排算法层
   - ops: 训练与运维层
   - common: 配置、日志、错误、响应等基础能力
2. 通过 Settings + config.json 管理运行参数，避免硬编码。
3. 使用 Blueprint 分拆 API，降低单文件耦合度。

### 2.2 推荐链路实现方式

1. Pipeline 统一编排，接口层仅做参数校验与上下文装配。
2. 召回阶段以向量检索为主，过滤已交互物品，输出候选集。
3. 排序阶段通过 MMoE 对候选进行多任务打分并聚合为最终分数。
4. 重排阶段执行轻量策略，保证结果可控和可替换。

### 2.3 训练与任务实现方式

1. 训练入口通过管理接口触发，任务进入线程执行器异步运行。
2. 同步写入数据库训练作业记录，并回写最终状态与指标。
3. 训练产物通过 ArtifactStore 统一登记，支持刷新加载与状态查看。

### 2.4 容错与稳定性实现方式

1. 对 MySQL/索引/特征缺失场景广泛使用 fallback，优先保障服务可用。
2. 日志中记录阶段事件与失败原因，支持快速定位问题。
3. 模型和索引加载提供降级路径，降低启动失败概率。

## 3. 当前问题与中期判断

### 3.1 明确存在的能力缺口

1. 搜索接口 /api/v1/search 仍为占位实现，尚未接入真实检索链路。
2. 文档中提到的部分可选策略（如多排序器灵活切换）与当前工厂实现存在偏差，当前固定为 two_tower + mmoe + random_shuffle。
3. 训练与推理的数据缺失策略尚未完全统一（部分点位偏 fail-fast，部分点位偏 fallback）。
4. 自动化测试目录在配置中已预留，但仓库内测试资产仍不充分，回归保障有限。

### 3.2 风险判断

1. 过度 fallback 可能掩盖数据质量问题，导致线上效果退化但不易被及时发现。
2. 异步任务当前为进程内线程模型，扩展到多实例部署时一致性与调度能力有限。
3. 召回/排序效果评估链路（离线指标面板、线上 A/B）尚未形成闭环。

### 3.3 中期结论

项目已完成从接口到算法到训练运维的主干闭环，具备可运行、可训练、可刷新、可观测的推荐服务基础。当前阶段重点应由“功能可用”转向“效果可证、稳定可控、工程可扩展”。
