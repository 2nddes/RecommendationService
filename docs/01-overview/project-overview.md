# Project Overview

## 文档目标

本文档面向第一次接触该项目的人，目的是回答四个最基础的问题：

1. 这个项目是做什么的。
2. 它解决了哪些问题。
3. 它的边界在哪里。
4. 现在这套实现已经具备哪些能力，还缺哪些能力。

如果你还不知道应该先看代码里的哪个文件，请先读完本文，再继续看 [architecture-overview.md](architecture-overview.md) 和 [repository-map.md](repository-map.md)。

## 1. 项目定位

RecommendationService 是一个以影视内容为主域模型的推荐后端服务。它把原本会分散在多个系统中的能力集中到了一个可运行的 Python 服务内，包括：

- 在线推荐
- 相似影片检索
- 热门榜计算与缓存
- 条件搜索
- RAG 问答
- 模型训练任务
- RAG 重建任务
- 启动预热与健康检查

从组织形态上看，它更接近“推荐中台后端”而不是“演示性质的机器学习脚本集合”。

## 2. 它在解决什么问题

### 2.1 不是只做单一模型推理

很多推荐项目只实现其中一段，例如：

- 只做召回模型训练
- 只做排序模型推理
- 只做搜索接口
- 只做向量问答 Demo

这个项目的重点不在某一个模型本身，而在于把多个推荐工程环节串成一个统一服务：

- 用统一 API 暴露多种能力
- 用统一配置管理在线与离线行为
- 用统一任务表管理训练与重建
- 用统一缓存层提升在线读性能
- 用统一运行态快照输出健康信息

### 2.2 让业务表直接参与推荐与检索

当前实现强依赖 MySQL 中的业务表，例如：

- `movie`
- `rating`
- `user_click`
- `user_collect_movie`
- `movie_comment`
- `movie_tag`
- `movie_person`
- `movie_embeddings`
- `ops_task`

这些表不是“旁路数据”，而是直接参与：

- 在线特征计算
- 召回候选生成
- 搜索过滤与排序
- RAG 证据构建
- 训练样本抽取
- 后台任务状态持久化

也就是说，这套系统的核心是“业务数据 -> 推荐 / 检索能力”的工程化打通。

## 3. 系统边界

### 3.1 这个项目负责什么

- 提供统一的 `/api/v1` REST / SSE 接口层。
- 提供在线推荐主链路。
- 提供相似影片与热门榜能力。
- 提供条件搜索能力。
- 提供 RAG 检索与流式中文回答。
- 提供训练任务与 RAG 重建任务入口。
- 提供运行状态观测与基础管理接口。

### 3.2 这个项目当前不负责什么

- 不负责前端页面。
- 不负责用户认证和权限体系的完整闭环，只暴露了面向内部或受控环境的管理接口。(推测)
- 不负责多服务部署编排，例如消息队列、专用任务调度器、分布式 tracing。
- 不负责完整的 CI/CD 交付说明。
- 不负责复杂的在线实验系统，例如灰度流量、A/B 平台、在线特征平台。(推测)

## 5. 核心能力全景

| 能力域 | 当前实现 | 说明 |
| --- | --- | --- |
| 用户推荐 | 已实现 | 通过召回、排序、重排生成个性化推荐结果，支持 paged / pop 两种缓存交付模式 |
| 相似影片推荐 | 已实现 | 优先复用 RAG embedding 检索，缺失时回退到 Two-Tower item 向量 |
| 热门榜 | 已实现 | 按时间窗口计算趋势分，支持 Redis 预热与回源计算 |
| 条件搜索 | 已实现 | 支持关键词、标签、日期、时长过滤和多种排序 |
| RAG 问答 | 已实现 | 基于 movie_embeddings + FAISS + 外部 LLM 提供 SSE 流式问答 |
| 模型训练 | 已实现 | 支持 Two-Tower、MMoE、XGBoost 训练任务 |
| 统一任务系统 | 已实现 | 使用 `ops_task` 作为训练与 RAG 重建的统一持久化任务表 |
| 健康检查 | 已实现 | 提供 `/health` 与 `/health/runtime`，输出 warmup、pipeline、rag、worker 状态 |
| 重排策略精细化 | 部分实现 | 当前只有 `RandomShuffleReranker`，更像占位版本 |
| 独立 worker 进程部署 | 未实现 | 当前采用进程内线程方式运行 worker |

## 6. 核心业务对象

### 6.1 影片

影片是整个系统的核心对象。无论是推荐、搜索还是 RAG，最终都围绕 `movie_id` 展开。

相关核心信息包括：

- 标题、年份、简介
- 评分总和与评分人数
- 收藏数
- 贝叶斯评分
- 标签
- 导演、演员
- 地区、语言

### 6.2 用户行为信号

系统当前主要消费以下行为信号：

- 点击：`user_click`
- 收藏：`user_collect_movie`
- 评分：`rating`
- 评论：`movie_comment`
- 兴趣标签：`user_collect_tag`

这些信号既参与在线推荐，也参与离线训练和特征构造。

### 6.3 任务

任务是这个仓库中很重要但容易被忽略的对象。训练、RAG 重建等后台动作不是“直接脚本执行”，而是经过统一任务表驱动。

这意味着任务本身也是系统的一等公民：

- 有任务类型
- 有状态流转
- 有输入 payload
- 有 progress 快照
- 有 result 输出

## 7. 运行模式概览

### 7.1 单体 Web 服务 + 内嵌后台线程

当前实现不是微服务，也不是“Web 和 Worker 完全拆开”的架构。服务启动后会：

- 初始化 Flask
- 注册蓝图
- 触发 warmup
- 加载推荐运行时
- 初始化 RAG 服务
- 启动后台线程

这是一个非常务实的结构：

- 本地开发简单
- 部署依赖少
- 适合课程项目、内部 PoC、单团队维护场景

但代价也很明显：

- 进程职责较重
- 启动时间较长
- worker 与 API 共进程，资源隔离较弱

## 8. 当前实现的几个重要事实

### 8.1 在线推荐主链路当前固定是 MMoE 排序

虽然仓库中也存在 XGBoost 排序训练逻辑，但在线组装 pipeline 时当前使用的是：

- Two-Tower Recall
- 可选 TagInverted Recall
- MMoE Ranker
- RandomShuffleReranker

因此不要把“仓库里能训练 XGB”误解为“线上默认用 XGB 排序”。

### 8.2 相似影片能力并不完全等同于推荐主链路

相似影片接口不是简单复用用户推荐 pipeline。它更偏向内容 / 向量相似性检索：

- 优先走 RAG embedding
- 再回退到 Two-Tower item vector

这使它成为一个独立的推荐能力，而不是用户推荐的变体。

### 8.3 搜索不是纯在线逻辑

搜索链路依赖一些预计算字段与索引，例如：

- `collect_count`
- `bayesian_rating`
- 搜索相关索引

因此搜索能力的稳定性不仅取决于 API 代码，还取决于后台预处理是否完成。

### 8.4 RAG 不是独立外挂功能

RAG 在这个项目中承担两类职责：

- 用户问答
- 相似影片向量检索

所以 RAG 故障的影响面并不只局限于聊天接口。

## 10. 项目当前最明显的维护重点

在继续迭代之前，建议优先关注这些问题：

1. 配置文件当前包含明文敏感信息，不适合直接对外交付。
2. 活跃模型路径、历史产物路径和仓库中已有模型目录之间存在潜在漂移，需要统一约定。
3. 重排策略目前较弱，只是随机打散。
4. RAG 和推荐 warmup 都在启动期同步执行，进程启动成本较高。
5. 文档还需要继续补齐推荐链路、搜索链路、RAG 细节、接口参考和运维 runbook。
